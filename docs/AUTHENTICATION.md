# Authentication

Skydeo Landing MCP is an internal service. Cloudflare Access is the membership
boundary, and the Worker exposes OAuth 2.1 endpoints to MCP clients through
`@cloudflare/workers-oauth-provider`.

## Fixed URLs

| Purpose | URL |
| --- | --- |
| Production MCP endpoint | `https://landing-mcp.skydeo.com/mcp` |
| Production OIDC callback | `https://landing-mcp.skydeo.com/callback` |
| Production previews | `https://*.landing-mcp.skydeo.com/<revision>` |
| Local MCP endpoint | `http://localhost:8787/mcp` |
| Local OIDC callback | `http://localhost:8787/callback` |

The exact production hostname is declared as a Custom Domain in
[`wrangler.jsonc`](../wrangler.jsonc). The route is configuration only until a real
deployment creates or updates the corresponding Cloudflare object.

## Request flow

1. An MCP client discovers the authorization metadata published by the Worker.
2. The Worker displays consent for the requesting MCP client.
3. The Worker redirects the browser to the Skydeo Access for SaaS application.
4. Cloudflare Access applies the Skydeo team policy and authenticates through the
   configured identity provider.
5. The callback verifies the Access ID token signature, expiry, required identity
   claims, and audience.
6. The Worker OAuth Provider issues an MCP-bound token containing the trusted
   Skydeo identity and granted landing scopes.
7. `LandingMcp` receives that context through `this.props`; tools must never accept
   user identity or permissions as authoritative tool input.

The upstream Access token is deliberately not copied into MCP session props. The
service currently needs identity, not delegated calls to another Access-protected
application.

## Separate signing-key endpoints

The MCP login and protected previews use different Access applications and must not
share a JWKS setting:

| Binding | Source | Verifies |
| --- | --- | --- |
| `ACCESS_JWKS_URL` | Key endpoint shown by the Access for SaaS OIDC application | Upstream identity tokens returned during MCP OAuth login |
| `PREVIEW_ACCESS_JWKS_URL` | `https://<team-name>.cloudflareaccess.com/cdn-cgi/access/certs` | Access assertions injected for the wildcard self-hosted preview application |

The preview audience is independently bound through `PREVIEW_ACCESS_AUD`. Mixing the
two key endpoints or audiences fails closed.

## 1. Create the Access application

In Cloudflare Zero Trust:

1. Go to **Access controls > Applications**.
2. Create a **SaaS application** using **OIDC**.
3. Name it `Skydeo Landing MCP`.
4. Add both callback URLs from the table above.
5. Attach the policy or reusable Access group that represents active Skydeo team
   members.
6. Select the company identity provider. If only one provider is allowed, enable
   instant authentication.
7. Copy the Client ID, Client secret, Token endpoint, Authorization endpoint, and
   Key endpoint shown for this Access application.

Use the values generated by Access for SaaS, not OAuth credentials copied from the
underlying Google, Microsoft, or other identity provider.

## 2. Create the preview Access application

Create a separate **self-hosted** Access application for
`*.landing-mcp.skydeo.com`. The wildcard covers generated Sandbox preview hosts but
does not cover `landing-mcp.skydeo.com`, so it cannot intercept the MCP server's own
OAuth endpoints.

Attach the same Skydeo team group used by the Access for SaaS application. Do not
use an `Include Everyone` or Bypass policy. Copy the self-hosted application's
**Application Audience (AUD) Tag**; it becomes `PREVIEW_ACCESS_AUD`.

Access authenticates the browser at the edge and injects a
`Cf-Access-Jwt-Assertion` header. The Worker verifies that assertion against
`PREVIEW_ACCESS_JWKS_URL` and `PREVIEW_ACCESS_AUD` before calling
`proxyToSandbox()`. `PREVIEW_ACCESS_JWKS_URL` must be the account-level Access
certificates endpoint at
`https://<team-name>.cloudflareaccess.com/cdn-cgi/access/certs`; it is distinct
from the Access for SaaS OIDC key endpoint used by the MCP login. A
missing, expired, not-yet-active, incorrectly signed, or wrong-audience assertion
fails closed.

## 3. Create OAuth storage

The OAuth Provider requires a Workers KV namespace for registered clients, grants,
tokens, and short-lived upstream state:

```sh
npx wrangler kv namespace create OAUTH_KV
```

Add the returned namespace ID to the existing `OAUTH_KV` entry in
[`wrangler.jsonc`](../wrangler.jsonc):

```jsonc
"kv_namespaces": [
  {
    "binding": "OAUTH_KV",
    "id": "<namespace-id>"
  }
]
```

Do not enable production Access mode until this ID is present.

## 4. Configure preview DNS and TLS

Sandbox `exposePort()` URLs require wildcard DNS and a Worker wildcard route in
production. In the `skydeo.com` zone, create this record:

| Field | Value |
| --- | --- |
| Type | `A` |
| Name | `*.landing-mcp` |
| IPv4 address | `192.0.2.0` |
| Proxy status | Proxied |

The Worker route `*.landing-mcp.skydeo.com/*` is already declared in
[`wrangler.jsonc`](../wrangler.jsonc).

These preview URLs use the second-level wildcard
`*.landing-mcp.skydeo.com`, which Cloudflare Universal SSL does not cover. Provision
an Advanced Certificate Manager or uploaded custom certificate that explicitly
covers that hostname before production verification.

## 5. Configure secrets

Required Worker secrets are declared in `wrangler.jsonc` so generated types remain
accurate:

```sh
npx wrangler secret put ACCESS_CLIENT_ID
npx wrangler secret put ACCESS_CLIENT_SECRET
npx wrangler secret put ACCESS_TOKEN_URL
npx wrangler secret put ACCESS_AUTHORIZATION_URL
npx wrangler secret put ACCESS_JWKS_URL
npx wrangler secret put PREVIEW_ACCESS_AUD
npx wrangler secret put COOKIE_ENCRYPTION_KEY
```

Generate the cookie encryption key with:

```sh
openssl rand -hex 32
```

For local authenticated MCP testing, copy [`.dev.vars.example`](../.dev.vars.example)
to `.dev.vars` and fill in the same seven values. `.dev.vars` is ignored by Git.

## 6. Test locally

The fast local development path remains intentionally unauthenticated and is only
bound to the local Wrangler server:

```sh
npm run dev
```

To exercise the complete Access browser flow instead:

```sh
npm run dev:access
```

The Access development script pins Wrangler's forwarded host to
`localhost:8787`. This keeps the OAuth issuer, authorization endpoints, and
protected-resource identifier on the local origin even though production uses a
configured Custom Domain. Loopback HTTP also uses development-only cookie names
without the `Secure` attribute; all non-loopback and production requests retain
strict `__Host-` cookies with `Secure` enabled.

Open MCP Inspector and connect to `http://localhost:8787/mcp` with Streamable HTTP.
Approve the MCP client, complete the Access login, then invoke
`get_service_status`.

## 7. Enable and verify production

Production uses `MCP_AUTH_MODE=access`. Before enabling any fresh environment,
verify all of the following:

- the Access policy allows only the intended Skydeo team group;
- both callback URLs are registered exactly;
- the wildcard self-hosted Access application uses the intended Skydeo team policy;
- `PREVIEW_ACCESS_AUD` is the audience of that wildcard application;
- wildcard DNS, Worker routing, and a certificate for
  `*.landing-mcp.skydeo.com` are active;
- the production `OAUTH_KV` namespace ID is committed in `wrangler.jsonc`;
- all seven Worker secrets are set;
- `npm run check` and `npm run deploy:dry-run` pass;
- publishing tools still require explicit, revision-bound confirmation.

Only then enable Access mode and perform an explicitly authorized deployment.

After deployment:

1. Confirm `/healthz` is reachable on the exact MCP hostname.
2. Connect MCP Inspector to `https://landing-mcp.skydeo.com/mcp`, complete login,
   and invoke `get_service_status`.
3. Create a draft and open its preview URL in a signed-out browser; Access must
   challenge rather than show the preview.
4. Sign in as an allowed Skydeo identity and confirm the exact revision renders.
5. Attempt the preview with an identity outside the policy and confirm denial.
6. Temporarily use a wrong `PREVIEW_ACCESS_AUD` in a non-production environment and
   confirm the Worker returns `401`, proving the origin-side check is active.

### Production smoke-test record

On 2026-07-22, the production deployment was verified end to end:

- the production MCP endpoint completed the Cloudflare Access OAuth flow;
- authenticated MCP calls reached the draft tools;
- an authenticated user opened the exact immutable Sandbox preview revision;
- a signed-out browser received the Cloudflare Access challenge instead of preview
  content; and
- the Worker-side assertion verifier accepted the preview application audience and
  removed identity credentials before forwarding to the Sandbox.

No access token, Access assertion, cookie, or tokenized preview hostname is retained
in this repository.

## Granted scopes

| Scope | Purpose |
| --- | --- |
| `landings:read` | Read draft metadata, validation status, and preview links |
| `landings:write` | Create drafts and request revisions |
| `landings:publish` | Confirm and request production publishing |

Access membership authenticates a Skydeo team member. Individual MCP tools must
still check the applicable scope before reading, changing, or publishing a draft.
Authentication alone does not make a preview available: after Access verification,
the Worker asks the owning `DraftCoordinator` for its lifecycle state. Expired,
revoked, and cleaned-up previews return `410` and are never proxied to the Sandbox.
